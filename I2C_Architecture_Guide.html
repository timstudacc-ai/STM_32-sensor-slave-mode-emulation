I2C Smart Sensor: Architectural Design Justification

This document details the firmware architecture for the STM32 I2C Slave and AVR Master, focusing on the failure modes of naive implementations and the contrastive reasoning behind our design choices.

1. The STM32 Slave: Non-Blocking State Machine

[Asynchronous Direction Routing]
// Why not this? (The Naive approach) while(1) { HAL_I2C_Slave_Receive(hi2c, buffer, 1, HAL_MAX_DELAY); if (buffer[0] == 0x06) { HAL_I2C_Slave_Receive(hi2c, &led_state, 1, HAL_MAX_DELAY); } }
[Blocking Deadlock Trap]

Why not use blocking HAL_I2C_Slave_Receive?

If we used a standard while-loop checking for incoming data using the blocking HAL_I2C_Slave_Receive function, the CPU would be entirely hijacked by the I2C peripheral.

The Failure Mode: If the Master unexpectedly disconnects mid-transaction, or if a clock pulse is lost due to EMI, the slave will wait forever at HAL_MAX_DELAY. The device hangs. Furthermore, while blocking, we cannot sample the ADC or debounce the button.

The Solution: The Mailroom Metaphor. Instead of waiting at the front door for the mailman (blocking), we use HAL_I2C_EnableListen_IT. This tells the hardware: "Ring the doorbell when someone asks for our address." The CPU is free to compute temperatures. When the address matches, the HAL_I2C_AddrCallback fires, and we dynamically route the incoming or outgoing bytes based on the TransferDirection.

[Stateful Payload Parsing]

What if we updated registers directly?

A naive implementation might take whatever byte arrives over the I2C bus and dump it directly into the memory map using a DMA channel or a blind pointer offset.

The Failure Mode: If the Master accidentally sends a pointer of 0x09 (which is out of bounds for our 8-byte array), a direct write would overwrite adjacent RAM on the STM32, causing a silent memory corruption fault. Alternatively, the Master could overwrite the DEV_ID read-only register.

The Solution: The Stateful Bouncer. We use a volatile flag g_is_first_byte. When the address matches, this flag is set to 1. The very first byte received is intercepted and assigned to g_reg_ptr. Any subsequent bytes in the same transaction are treated as data payloads. Before saving the payload, we enforce strict bounds checking (< g_reg_size) and hardcoded access rights (only LED_CTRL is allowed to be written). The pointer then auto-increments, allowing safe, continuous burst-writes.

// Burst Transmission & Boundary Safeties void HAL_I2C_SlaveTxCpltCallback(...) { // Auto-increment pointer after successful TX if (g_reg_ptr < g_reg_size) { g_reg_ptr++; } // Prepare next byte for Master's ACK if (g_reg_ptr < g_reg_size) { HAL_I2C_Slave_Seq_Transmit_IT(..., ®_file[g_reg_ptr], 1, ...); } else { // Prevent Memory Leak! uint8_t dummy = 0xFF; HAL_I2C_Slave_Seq_Transmit_IT(..., &dummy, 1, ...); } }
[Sequential Memory Streaming]

Why auto-increment?

If we forced the Master to send a new register pointer for every single byte it wanted to read, we would waste significant bandwidth on repeated [Start] and [Address] overhead. By auto-incrementing g_reg_ptr inside the TX complete callback, we enable Burst Reads. The Master sends the pointer once (e.g., 0x02 for Temperature), and then simply clocks in 3 bytes sequentially. The slave automatically serves 0x02, 0x03, and 0x04 on the fly.

What if the Master reads out of bounds?

The Failure Mode: If the Master asks for 10 bytes but our register map is only 8 bytes long, a naive implementation would blindly increment the pointer and stream whatever data resides in RAM physically adjacent to our array. This is a severe vulnerability (Data Leakage) and can cause unpredictable application behavior.

The Solution: The Dummy Byte. We check g_reg_ptr < g_reg_size before queueing the next byte. If the Master gets greedy and reads past the end of the array, we intentionally clamp the pointer and stream a harmless 0xFF dummy byte instead of exposing random memory.

2. The AVR Master: Timeout-Protected Bare Metal

// Custom TWI start with timeout static TWI_Status_t TWI_WaitForComplete(void) { uint32_t start_time = Tick_GetMs(); while (!(TWCR & (1 << TWINT))) { if ((Tick_GetMs() - start_time) > TWI_TIMEOUT_MS) { return TWI_ERROR_TIMEOUT; } } return TWI_OK; }
[Hardware State Polling]

What if we only used Wire.h?

The standard Arduino Wire library uses infinite while-loops internally. If an I2C slave loses power mid-transaction while holding the SDA line low, the master's Wire.endTransmission() will loop infinitely. Your entire robot or system freezes because a $2 sensor came unplugged.

The Solution: The Chain Reaction. We wrote a bare-metal register driver for the ATmega328P. Every interaction with the TWCR control register is wrapped in TWI_WaitForComplete(), which utilizes our Timer0 1ms tick. If TWINT doesn't set within 100ms, we abort and return a TWI_ERROR_TIMEOUT.

Why Bit-Bang Recovery?

If a timeout occurs, dropping the transaction isn't enough. The slave might still be holding SDA low, waiting for clock pulses. We execute a Bus Recovery sequence: we temporarily convert the SCL and SDA pins to standard outputs, and manually clock SCL 9 times. This forces the confused slave hardware state machine to eject any pending bits and release the bus.

Deep Dive: Master Read & Write Protocols

Interacting with a register-mapped I2C device requires specific byte sequences to ensure the slave knows which register we want to access.

The Write Sequence: To write to a specific register (like LED_CTRL), the Master first sends a START condition, followed by the Slave Address with the Write bit (0). The next byte sent is the Register Pointer (reg_addr). The slave catches this byte and updates its internal pointer. Finally, the Master sends the actual data payload to be written to that register, and concludes with a STOP condition.

The Read Sequence (Repeated Start): Reading data requires a change in bus direction mid-transaction. First, the Master pretends it's going to write by sending the Slave Address (Write = 0) and the Register Pointer. Then, instead of sending data, it issues a Repeated START condition. This keeps the bus claimed but resets the state machine. The Master then sends the Slave Address again, but this time with the Read bit (1). The slave, remembering the pointer from earlier, begins streaming data. The Master ACKs every byte except the final one, which it NACKs to tell the slave to release the line.

The "Gotchas"

Critical hardware quirks discovered during implementation that cause silent failures.

Voltage Tolerance Mismatch

The ATmega328P drives I2C lines to 5V via its internal pull-ups. The STM32F411 runs at 3.3V. While some pins on the STM32 are "5V tolerant", directly connecting them without checking the specific datasheet pinout (like PB6 and PB7) can blow the GPIO driver. Silent Failure: The pin permanently reads as logic LOW.

STM32 I2C Clock Stretching Deadlocks

If you put a breakpoint inside HAL_I2C_SlaveRxCpltCallback while debugging the STM32, the hardware automatically holds the SCL line LOW (Clock Stretching). Because the Master's TWI_TIMEOUT_MS is only 100ms, the Master will timeout and attempt a bus recovery while you are still staring at your debugger. Silent Failure: Debugging causes phantom bus errors.

IWDG vs WWDG Confusion

If you accidentally select the Window Watchdog (WWDG) instead of the Independent Watchdog (IWDG) and kick it inside a fast loop, the MCU will reset instantly. WWDG requires you to wait for a specific time window to open before refreshing. Silent Failure: Instant, continuous boot-looping.
